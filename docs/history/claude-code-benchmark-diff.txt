● 性能影响分析

  速度

  ┌─────────────────────────────┬───────────────────────────────────────────┐
  │            变更             │                   影响                    │
  ├─────────────────────────────┼───────────────────────────────────────────┤
  │ per_sequence_quality        │ 微快 — 去掉了 rounding 操作               │
  │ .round() → as i32           │                                           │
  ├─────────────────────────────┼───────────────────────────────────────────┤
  │ format_double 科学计数法    │ 零影响 — 仅在报告生成时执行，不在热路径   │
  ├─────────────────────────────┼───────────────────────────────────────────┤
  │ OverRepresentedSeqs 移到    │ 零负面影响 — reader 线程原来在等          │
  │ reader 线程                 │ I/O，现在顺便做 dedup 追踪                │
  └─────────────────────────────┴───────────────────────────────────────────┘

  之前的共享 Mutex 方案会让 4 个 worker 每条序列都竞争锁（~10% 串行化 → Amdahl's
   law 预估从 4x 降到 ~3.1x）。现在的方案完全没有锁竞争 — OverRepresentedSeqs 在
   reader 线程顺序执行，worker 线程完全并行处理其他所有模块。

  Peak Memory

  ┌──────┬───────────────────────────────────────────────────────┐
  │ 变更 │                         影响                          │
  ├──────┼───────────────────────────────────────────────────────┤
  │ 之前 │ 4 worker × 各自 100K unique sequence HashMap ≈ 4份    │
  ├──────┼───────────────────────────────────────────────────────┤
  │ 现在 │ 1 个 reader 线程 × 100K unique sequence HashMap ≈ 1份 │
  └──────┴───────────────────────────────────────────────────────┘

  Peak memory 降低约 75%（针对 dedup 数据结构部分）。Worker 仍然各自有独立的
  OverRepresentedSeqs（在 create_module_list 中创建），但这些 worker 的 dedup
  数据最终被丢弃替换为 reader 的正确版本。

  如果要进一步优化内存，可以让 worker 的模块列表完全不包含
  OverRepresentedSeqs/DuplicationLevel，但这需要更大的重构（模块索引匹配问题）。
  当前方案是正确性和简洁性的最佳平衡。
