  关键发现：

  1. CPU 计算时间几乎不变 (~48-50s user)，说明计算量是固定的
  2. Wall time 差了 2-3 倍，因为线程在等磁盘 I/O
  3. CPU 利用率从 500% 降到 200-300%，说明多线程在等数据，大量时间 idle

  结论：热缓存时瓶颈在 CPU 计算；冷缓存时瓶颈在磁盘读取（1.15GB gz
  文件）。实际生产场景中文件通常不在缓存里，所以 真实速度更接近 16-25s 而非 
  9.5s。

● 基于三次测试数据来估算：

  估算磁盘读取速度

  文件大小：1.15 GB (gz 压缩)

  ┌────────────┬────────┬────────────────┬─────────────────────────┐
  │            │ 热缓存 │ 冷缓存 (cargo) │ 冷缓存 (GitHub Actions) │
  ├────────────┼────────┼────────────────┼─────────────────────────┤
  │ Wall time  │ 9.5s   │ 24.9s          │ 16.4s                   │
  ├────────────┼────────┼────────────────┼─────────────────────────┤
  │ CPU time   │ ~48s   │ ~50s           │ ~48s                    │
  ├────────────┼────────┼────────────────┼─────────────────────────┤
  │ CPU 利用率 │ 500%   │ 203%           │ 299%                    │
  └────────────┴────────┴────────────────┴─────────────────────────┘

  计算逻辑：热缓存时 I/O 约等于 0（从内存读取），所以纯计算 wall time ≈
  9.5s。冷缓存多出的时间主要花在磁盘 I/O 上（I/O 和计算有部分重叠）：

  cargo build 版本：
  - I/O 额外耗时 ≈ 24.9 - 9.5 = 15.4s（上界，假设完全无重叠）
  - 压缩数据吞吐：1.15 GB / 15.4s ≈ ~75 MB/s
  - 如果 I/O 是瓶颈（完全重叠）：1.15 GB / 24.9s ≈ ~46 MB/s

  GitHub Actions 版本：
  - I/O 额外耗时 ≈ 16.4 - 9.5 = 6.9s
  - 压缩数据吞吐：1.15 GB / 6.9s ≈ ~167 MB/s
  - 如果 I/O 是瓶颈：1.15 GB / 16.4s ≈ ~70 MB/s

  估计实际磁盘读取速度在 50-75 MB/s 左右（WSL2 环境）。

  两个二进制差异很大（24.9s vs 16.4s），GitHub Actions 版本快了 50%。可能原因：
  1. 编译目标不同（GitHub Actions 可能启用了更激进的 target-cpu 优化）
  2. gzip 解压效率差异（这也是 I/O 链路的一部分）

  你可以跑 lscpu 和对比下两个二进制的编译参数，看看是否 GitHub Actions 版开了
  target-cpu=native 或 AVX 指令集之类的。

● 找到原因了。对比结果：

  两个二进制的差异

  ┌──────────────┬─────────────────┬───────────────────────────────────────┐
  │              │  cargo build    │            GitHub Actions             │
  │              │     (本地)      │                                       │
  ├──────────────┼─────────────────┼───────────────────────────────────────┤
  │ Rust 版本    │ 1.89.0          │ 更新版本 (e408947b)                   │
  │              │ (2025-08-04)    │                                       │
  ├──────────────┼─────────────────┼───────────────────────────────────────┤
  │ 编译命令     │ cargo build     │ cargo build --release --target        │
  │              │ --release       │ x86_64-unknown-linux-gnu              │
  ├──────────────┼─────────────────┼───────────────────────────────────────┤
  │ 运行环境     │ 本机 i9-13900K, │ GitHub runner (不同 CPU)              │
  │              │  WSL2           │                                       │
  ├──────────────┼─────────────────┼───────────────────────────────────────┤
  │ Wall time    │ 24.9s           │ 16.4s                                 │
  │ (冷缓存)     │                 │                                       │
  └──────────────┴─────────────────┴───────────────────────────────────────┘

  但关键点是：两个都没有开启 target-cpu=native。你的 i9-13900K 支持
  AVX2、AVX-VNNI、SHA-NI 等高级指令集，但默认编译只用了基础的 SSE2。

  速度差异更可能来自 Rust 编译器版本不同（GitHub Actions 用了更新的
  rustc，可能有更好的代码生成优化），而不是指令集差异。

  建议：在 Cargo.toml 或编译时加上 RUSTFLAGS="-C target-cpu=native"
  可以充分利用你 CPU 的 AVX2 指令集，对 gzip 解压和数据处理可能有明显提速。
